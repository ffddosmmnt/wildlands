  The repo is a fresh, empty Godot 4.7 project вҖ” just the default icon and empty

  The deliverable

  This is a planning-only task. The one output is a design document вҖ”
  documents/IMPLEMENTATION_PLAN.md (TASK.md said docs/, but your repo uses

  The architecture I'm recommending

  Eight foundational decisions, in priority order:
  1. Flyweight вҖ” shared .tres Definitions vs unique runtime Instances. This is
  what makes both "1000 items" and "Bone #001 вү  Bone #002" work, and it dictates
  the save strategy.
  2. Database autoload вҖ” scans res://data/**, so adding content = adding a 
  .tres, not code.
  3. EventBus autoload вҖ” decoupled signals; the backbone of learn-by-doing.
  4. Generic property + formula system вҖ” the Durango heart; recipes reference
  property keys, never material names.
  5. Tag-based crafting slots вҖ” sharp_material + handle_material, not raptor 
  bone + acacia.
  6. Composition for combat & AI вҖ” one CombatStats component + behavior modules;
  no per-creature scripts.
  7. Save = instances only (definitions rehydrate from .tres).
  8. Isometric rig + placeholders вҖ” visuals last, cubes throughout V1.

  The roadmap

  13 system-first milestones (M0вҖ“M12), each ending in something runnable and 
  verifiable. The critical one is M7, where the full core loop first closes:
  craft spear вҶ’ hunt herbivore вҶ’ get bone вҶ’ craft better spear вҶ’ fight predator.
  M5's verification specifically proves material-driven crafting (same recipe,
  different materials вҶ’ different item). Everything maps back to the
  ADDITIONAL_TASK acceptance checklist; out-of-scope items are allowed-for but
  not scheduled.

  One deliberate simplification worth flagging: the milestone list reorganizes
  TASK.md's Phases 0вҖ“11 into buildable, testable chunks rather than following
  them 1:1 вҖ” same content, ordered so each step runs.
